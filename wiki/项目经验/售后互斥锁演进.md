# 售后互斥锁演进

## 一句话介绍
负责售后中心互斥锁方案的多轮演进，围绕 `Redis SETNX + TTL` 在超时重试、自锁、复用锁、补寄兼容、主从延迟等问题上的缺陷持续治理，把“能拦并发”升级成“可靠拦并发且尽量不误伤业务”。

## STAR 自陈
- **S（场景）**：售后申请、修改售后、审核、取消仲裁、编辑物流等操作之间存在大量互斥关系，而且链路横跨多个 PSM。只要互斥锁不可靠，就会出现并发提单、状态异常、同一单据多个互斥操作同时执行等问题。
- **T（目标）**：建设一套稳定可用的售后互斥锁能力，既能拦住真实并发冲突，又要降低因为 Redis 抖动、重试、主从延迟、上下游链路传参不一致带来的误判和兼容问题。
- **A（动作）**：
  1. 梳理售后互斥场景和现网锁模型，明确锁 key、锁 value、TTL、加锁/解锁时机以及复用锁链路。
  2. 将原始 `redisClient` 升级为带重试能力的 `retryredis`，降低 `setnx` 超时导致的锁可靠性问题。
  3. 针对“超时成功”导致的自锁问题，引入全局唯一 `lockVal`，并在 `SETNX` 返回失败时校验 Redis 中的锁值，识别“其实是自己前一次请求加锁成功”的场景。
  4. 为兼容补寄链路，在锁结构体中显式记录 `lockVal`，并在 workflow 后置节点手动把锁信息写回 `ctx`，让下游 PSM 能正确复用锁。
  5. 继续推进锁粒度拆分，把申请类操作和其他售后操作区分为 `orderId` 锁 / `afterSaleId` 锁，减少“先查再锁”带来的脏数据和并发窗口问题。
  6. 识别主从延迟带来的复用锁失败、加锁成功但返回失败问题，推动后续读主方案和更稳妥的兜底设计。
- **R（结果）**：
  - 将 `setnx` 超时错误率从约 `0.04%` 降到约 `0.0008%`，显著提升互斥锁可用性。
  - 在 BOE `10000` 次压测中定位出 `5` 次“自己锁住自己”的问题，并通过后续版本修复到“基本不影响接口成功率”。
  - 完成补寄场景兼容，降低了多链路复用锁失败带来的业务回滚风险。
  - 在多轮版本演进中沉淀出上线评估重点：兼容老版本锁值、关注共存期、强化回归测试与异常链路验证。

## 核心问题
- **V1 问题**：`SETNX` 一旦超时或报错，代码直接降级为“加锁成功”，互斥能力不可靠。
- **V2 问题**：引入重试后，解决了大部分超时失败，但出现“第一次请求其实成功、第二次重试失败，客户端误以为锁被别人抢走”的自锁问题。
- **V3 问题**：通过唯一 `lockVal` 解决自锁后，又暴露出补寄链路无法透传锁信息、下游无法复用锁的问题。
- **V4 问题**：兼容补寄后，继续暴露出锁粒度不合理、加锁时机偏晚的问题。
- **V5/V6 问题**：申请类和非申请类接口对锁 key 的依赖不同；同时 Redis 主从延迟还会影响锁值读取和复用判断。

## 核心方案

### 1. 锁模型
- 锁 key：早期统一使用 `aftersale:op_lock:${orderId}`
- 锁 value：从 `psm_opId_logId` 演进为带唯一 ID 的 `lockVal`
- 锁 TTL：默认 `10s`
- 复用锁：上游已持锁时，下游沿调用链复用同一把锁，而不是再次抢锁

### 2. 关键设计
- **重试不是终点**：仅靠重试可以降低错误率，但不能解决“超时成功”的语义不一致问题。
- **锁值必须唯一**：只有给每次加锁请求分配唯一 `lockVal`，才能判断失败到底是别人持锁，还是自己前一次请求已成功。
- **释放锁要认主**：文档也指出早期释放锁没有校验锁归属，存在误删别人锁的风险，这是面试里可以主动补充的改进点。
- **复用锁依赖链路透传**：补寄问题本质不是 Redis 锁本身，而是 workflow 节点和下游 PSM 之间的上下文传递不完整。
- **兼容比改代码更难**：锁值格式变化、锁 key 变化、加锁时机前移，都会遇到新老版本共存的问题，必须把上线顺序和回归测试纳入方案设计。

## 版本演进
1. **V1**：原始 `SETNX` 方案，报错降级为成功，互斥不稳定。
2. **V2**：接入 `retryredis`，把错误率降到 `0.0008%` 左右，但引入自锁问题。
3. **V3**：引入唯一 `lockVal`，通过回读 Redis 锁值识别“自己前一次请求已成功”。
4. **V4**：锁结构体持有 `lockVal`，并在 workflow 后置节点补写 `ctx`，兼容补寄场景。
5. **V5**：按申请类 / 非申请类拆分锁粒度，逐步推进 `orderId` 锁与 `afterSaleId` 锁并存改造。
6. **V6**：继续处理主从延迟带来的读写不一致问题，规划读主与更强兜底方案。

## 这个项目怎么讲
可以按下面这条线展开：
1. **问题背景**：售后链路互斥操作很多，一个锁问题会直接体现在状态错乱和接口成功率波动上。
2. **关键难点**：不是简单“上 Redis 锁”就结束，真正难的是超时重试、自锁、复用锁、上下游透传、主从延迟和新老版本兼容。
3. **核心设计**：用唯一 `lockVal` 区分“别人持锁”和“自己前一次请求成功”，并把复用锁能力纳入上下游链路设计。
4. **工程落地**：从客户端重试、锁值设计、workflow 透传、锁粒度拆分、上线兼容、回归测试几个维度持续演进。
5. **结果收口**：用错误率下降、BOE 压测定位、自锁修复、补寄兼容、两次回滚经验这些事实证明这不是“纸面方案”，而是真正在线上打磨过的稳定性项目。

## 可追问点
- 为什么 `retryredis` 不能彻底解决互斥锁问题？
- 什么叫“超时成功”，为什么会把自己锁住自己？
- 为什么要给每次加锁请求生成唯一 `lockVal`？
- 复用锁和普通抢锁的区别是什么？
- 为什么锁粒度要从统一 `orderId` 锁，演进到 `orderId` / `afterSaleId` 分组？
- Redis 主从延迟会怎样影响分布式锁正确性？
- 如果让你继续做，释放锁、续期、读主、MySQL 兜底会怎么设计？
