# 退款成功率优化

## 一句话介绍
围绕售后退款链路做一次端到端成功率治理，从离线数据洞察、系统内部重试优化、财经错误码标准化推动到支付渠道结构调整，重点解决“退款不到账、优惠券未退回”等用户感知问题，把 `24h` 退款成功率从 `99.754%` 持续往 `4` 个 `9` 推进。

## STAR 自陈
- **S（场景）**：当时退款相关用户进线很多，典型问题是“为什么退款还没到账”“为什么优惠券没退回”。对退款数据做洞察后发现，`24h` 退款成功率只有 `99.754%`，量级放大后意味着会有大量长尾失败或长耗时订单。
- **T（目标）**：围绕退款时效和退款成功率做专项优化，核心目标是把 `24h` 成功率从 `99.754%` 提升到接近 `99.999%`，同时找出为什么 `1min` 只能做到 `99.8%` 左右，以及哪些环节决定了进一步优化空间。
- **A（动作）**：
  1. 按 `超过24h退款成功 / 超过24h退款失败 / 超过24h退款中` 三类做离线数据拆解，定位最耗时、最容易卡住的链路。
  2. 在退款系统内部梳理各节点耗时问题，优化异常退避重试和补偿逻辑。
  3. 推动财经侧对齐成功率目标，拉齐错误码和错误原因透出，减少只能靠 `T+1` 补偿的场景。
  4. 结合支付渠道差异分析退款时效瓶颈，推动提升抖音支付占比，并把部分退款失败链路改造成自动退转付。
- **R（结果）**：
  - 系统内部优化后，退款成功率提升约 `0.1%`。
  - 退款失败转赔付链路从“每天 `4000~5000` 单人工处理”逐步优化到 `T+1` 批处理，再到实时退转付，最终降到每天约 `100+` 单。
  - 识别出进一步提升的主要瓶颈不在电商退款系统本身，而在财经回调慢、退款失败转赔付慢，以及部分支付渠道自身退款时效。

## 问题拆解
- **用户感知问题明显**：退款没到账、优惠券没退回，都会直接变成进线投诉。
- **链路长，责任分散**：售后、电商支付、财经、支付渠道都可能引入额外耗时。
- **错误码不透明**：像“等待资金归集完成”这类错误，在部分节点没有标准透出，导致系统无法即时重试，只能依赖 `T+1` 补偿。
- **天级补偿存在漏补场景**：极速退两阶段退款中，第一阶段平台退用户成功发起，但由于补偿逻辑只覆盖售后完结态，导致部分单子会隔多天才被重试。
- **渠道天花板客观存在**：不同支付渠道退款时效差异很大，尤其小银行覆盖多的渠道更容易拖低整体 `24h` 成功率。

## 核心方案

### 1. 先做数据洞察，再做定点治理
- 不是笼统讲“退款成功率低”，而是拆成三类：
  - 超过 `24h` 才成功
  - 超过 `24h` 仍失败
  - 超过 `24h` 仍处理中
- 再按退款状态、支付渠道、具体节点去看耗时分布，判断到底是系统重试慢、财经回调慢，还是渠道本身慢。

### 2. 退款系统内部优化
- **异常重试策略优化**：
  - 原来退避重试节奏偏慢，如 `5min`、`10min`
  - 调整为更激进的即时重试节奏，如 `10s`、`1min`、`2min`
  - 这类优化直接改善了售后退款和异常恢复时效
- **补偿脚本优化**：
  - 原有天级补偿只覆盖售后单完结态，遗漏了极速退两阶段中的部分场景
  - 针对售后 `11` 状态补齐补偿逻辑，避免某些资金单线上请求失败后要等很多天才被重新处理

### 3. 推动财经侧优化
- 和财经同学对齐退款成功率目标，而不是把它当成单方系统指标。
- 推动财经支付侧错误码标准化，要求明确透出错误原因，让上游系统能做即时重试，而不是只能等 `T+1`。
- 对退款失败场景按原因做分流处理：
  - **保证金不足**：通过风控后支持平台垫资退
  - **用户账户问题**：部分订单支持退转付
  - **其他异常**：未命中风控则转客服人工赔付，命中风控则继续卡控

### 4. 支付渠道治理
- 对比支付宝、微信、抖音支付等渠道在 `1min / 5min / 10min / 24h` 的退款成功率表现，识别真正限制天花板的渠道。
- 结论上，微信因为覆盖银行更多，尤其小银行较多，退款时效天然更慢，整体表现弱于支付宝。
- 后续策略不是单纯继续抠系统代码，而是通过渠道结构调整来拉升整体成功率，比如降低微信支付占比，推动抖音支付。

### 5. 退款失败转赔付链路优化
- 原来大量退款失败单需要客服人工赔付，效率低，用户体验差。
- 优化路径分三步：
  1. 无策略，人工处理
  2. `T+1` 退转付，按次日离线批处理
  3. 实时退转付
- 这类改造把每天数千单人工处理规模显著压缩到百单级别。

## 结果与收益
- `24h` 退款成功率基线是 `99.754%`，专项优化后有显著提升。
- 仅系统内部优化就带来约 `0.1%` 的提升。
- 退款失败转赔付场景从每天 `4000~5000` 单人工处理，下降到每天约 `100+` 单。
- 通过这次治理，团队对退款问题形成了更清晰的拆解框架：
  - 系统内部问题
  - 财经协同问题
  - 支付渠道时效问题
- 也明确了后续优化边界：当系统和财经问题被解决后，剩余瓶颈主要来自支付渠道本身。

## 这个项目怎么讲
可以按下面这条线讲：
1. **先讲问题规模**：用户进线很多，`24h` 退款成功率只有 `99.754%`，放大到大盘就是很大的真实问题。
2. **再讲方法**：不是拍脑袋优化，而是先按“超 `24h` 成功 / 失败 / 处理中”做离线数据洞察。
3. **接着讲落地动作**：内部重试优化、补偿脚本修复、推动财经错误码标准化、退款失败转赔付自动化、渠道结构优化。
4. **最后讲认知升级**：这个项目的价值不只是提了一个指标，而是把退款成功率问题从“单系统问题”变成了“链路级治理问题”。

## 可追问点
- 为什么先按“超 `24h` 成功 / 失败 / 处理中”来拆，而不是直接看整体成功率？
- 为什么有些场景只能做 `T+1` 补偿，根因是什么？
- 退款系统内部优化为什么只能提升约 `0.1%`，后续瓶颈在哪里？
- 为什么微信渠道会比支付宝更难优化？
- 退款失败转赔付从人工到实时退转付，中间最大的工程难点是什么？
