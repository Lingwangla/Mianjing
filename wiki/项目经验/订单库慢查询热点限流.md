# 订单库慢查询热点限流

## 一句话介绍
针对订单查询链路 `500+` 接入方、整体 `400w+ qps` 的场景，设计并上线“后置热点检测 + 分布式封禁式限流”方案，按 `上游链路 + 表名 + 冷热库 + user_id + 点查/列表查` 做细粒度隔离，把热点请求压到个位数 qps，避免慢查询打挂 DB 并波及其他业务。

## STAR 自陈
- **S（场景）**：订单查询链路（`ecom.trade.order_query`、`ecom.trade.order_center`）是通用查询入口，上游接入方非常多。一旦某个上游出现聚集性慢查询，就可能把从库分片连接打满，导致整条订单查询链路受影响。
- **T（目标）**：建立细粒度热点限流能力，做到只打掉问题流量，不误伤其他接入方；同时补齐链路、冷热库、热点维度的监控，为后续治理提供数据。
- **A（动作）**：
  1. 设计热点 key：按 `源头 PSM + 表名 + 冷热库 + user_id + 查询类型` 维度限流，尽量在最源头隔离问题流量。
  2. 设计 caller 获取逻辑：优先取 `Base.Extra.user_extra.RPC_PERSIST_ori_psm`，兜底 `routekitex.GetRealCaller()`，避免只按中间业务层 caller 限流导致整层误伤。
  3. 采用“后置热点检测 + 封禁”方案：不是所有请求都走前置分布式限流，而是慢查询发生后上报热点，再针对命中的热点做分布式限流。
  4. 在 `TSF limiter`、`order_domain`、`order_model` 等模块补齐热点上报与命中限流逻辑。
  5. 建立监控、日志定位、初始化配置、阈值调节 SOP，并在 BOE / PPE 做多场景自测。
- **R（结果）**：
  - 线上已多次拦截真实热点流量，典型如 `2024.3.6` 单用户 `2k+ qps` 的物流刷单场景，限流持续 `17:17~17:42`，数据库虽有线程尖刺但未影响整体可用性。
  - `2026.1.29` 类目标项目更新打标逻辑热点中，方案拦截了 `84` 个分片上的热点慢查询，虽然仍有部分主库受损，但已显著缩小影响面，并给后续调参提供了抓手。

## 为什么不能直接用普通限流
- 订单查询是共享基础链路，不能简单按上游 `psm + 集群` 粗暴限流，否则会把正常业务一起打掉。
- 如果所有请求都走前置分布式限流，需要大量 redis 资源，而且每次查询额外增加 `5~10ms`，对 `< 50ms` 的查询接口不可接受。
- 数据库超时 kill 太晚，慢 SQL 并发持续时仍可能先把连接打满。
- 单分片限流也不够，因为它识别不了“低 qps 但单次极慢”的慢查询。

## 核心方案

### 1. 限流维度
- 粒度：`上游链路 + 表名 + 冷热库 + user_id + 点查/列表查`
- 核心考虑：限流 key 越细，对其他上游的隔离效果越好。
- 难点：真实调用链上经常会经过 `trade_order_manage`、`shop.order_query` 这类业务层服务，所以不能只看表面 caller，要尽量还原源头 PSM。

### 2. 限流流程
整体思路不是“前置整形”，而是“后置封禁”：
1. 请求正常访问订单库。
2. 如果查询达到慢查询阈值，则上报热点平台。
3. 在滑动窗口内累计达到热点阈值，判定该 key 为热点。
4. 后续命中该热点 key 的请求，再走分布式限流。

这种设计适合“总流量巨大，但真正有问题的热点只占极小部分”的场景，本质上是在大流量里做精确拦截。

### 3. 与其他方案的区别
- **相比数据库超时 kill**：更早，在 SQL 落 DB 前拦截。
- **相比数据库单分片限流**：能识别慢查询热点，而不是只看分片整体流量。
- **相比提下单热点限流**：提下单更像“流量整形”，而这里更像“大海捞针”，只对“慢查询且存在热点”的流量做分布式限流。

## 关键配置
- **慢查询阈值**：
  - 点查 `oid_ms = 500ms`
  - 列表查 `uid_ms = 1000ms`
- **热点判定**：
  - 滑窗总时长 `20000ms`
  - 滑窗数 `40`
  - 热点阈值 `10`
  - 热点过期时间 `300000ms`
  - `limit_n = 200`
- **限流阈值**：
  - 热点命中后限到 `10 qps`

通常线上主要调的是**限流阈值**；如果热点覆盖不够，再考虑调热点阈值、过期时间和 `limit_n`。

## 验证与上线
- 相关改动覆盖 `Infrastructure`、`TSF`、`order_domain`、`order_query`、`order_model`、`order_center` 多个模块，文档中有对应 CR 记录。
- BOE / PPE 自测覆盖了多类接口与场景：
  - 未达慢查询阈值：不上报热点
  - 达到慢查询阈值但未达热点阈值：只上报，不限流
  - 达到热点阈值：下次同热点请求被限流
  - 其他用户不受影响
  - 热点过期后恢复
  - 热点过多时仅下发 TOPN 热点
- 上线前后还补了 `user_id = 0` 的兼容逻辑，避免打标查询异常影响限流判断。

## 线上排障 SOP
### 监控
- 重点关注三个指标：慢查询 QPS、命中热点 QPS、被限流 QPS。
- 监控入口是交易容灾 / MySQL 相关大盘。

### 热点定位
- 报警出现后，用报警里的 `user_id` 去搜日志。
- 日志里关注 `origin` 字段，定位真实源头 PSM，再找对应业务方确认根因。

### 调参
- 默认先调限流阈值；
- 如果发现热点拦截不够，才进一步调整慢查询阈值、热点阈值、热点过期时间和 `limit_n`。

## 这个项目怎么讲
可以按下面这条线展开：
1. **问题背景**：共享订单查询链路，接入方多，慢查询会放大成全局问题。
2. **核心难点**：不能粗暴按 caller 限流，也不能让所有请求都走 redis 分布式限流。
3. **关键设计**：后置热点检测 + 封禁式限流，只对真正有问题的热点 key 下手。
4. **工程落地**：热点 key 设计、源头 caller 还原、多模块接入 limiter、监控和调参 SOP。
5. **结果证明**：线上多次成功拦截真实事故流量，做到“问题流量被压住，其他业务尽量无损”。

## 可追问点
- 为什么热点 key 里要带 `user_id`，会不会导致 key 数量膨胀？
- 为什么前置 redis 限流不适合这个场景？
- 如果热点不是单用户，而是某类公共查询语句怎么办？
- `limit_n` 为什么要大于热点 server broker 数？
- 2026.1.29 那次为什么已经有限流，仍然有主库被打挂？后续怎么继续优化？
