## 重点项目

### 1. 后返业务逆向标准流程建设
- **背景**：买后返、以旧换新、超市卡后返等营销玩法在退款场景下需要追缴营销资产，否则容易被用户套利；同时又要保证用户申请页“所见即所得”。
- **难点**：
  - 追缴结果依赖营销资产实时使用状态，申请前后结果可能变化。
  - 后返逻辑渗透申请页、提单、修改售后、换货转退款、审核、退款等多个节点。
  - 不同后返活动存在定制规则，且活动间还可能叠加。
- **方案**：
  - 在售后提单阶段引入渲染结果变更比对。
  - 设计冻结、解冻、核销流程，保证提单后执行结果与用户感知一致。
  - 将后返流程标准化为：`咨询 -> 冻结 -> 核销 -> 解冻 -> 退款扣减 -> 追缴分账 -> 平账`。
  - 抽象标准化数据模型、交互协议和追缴优先级，开放定制点支撑不同后返活动差异化规则。
- **收益**：
  - 可以低成本快速支持新的后返玩法。
  - 在逆向流程中内建追缴能力，降低平台和商家的营销资损风险。

### 2. 退款系统建设与交易模式升级
- **背景**：退款流程中平台与业务逻辑耦合严重，代码重复多，查询和模型设计面向过程，难以支撑后续复杂业务扩展。
- **方案**：
  - 设计逆向交易整体分层架构，推动平台与业务分离、业务与业务隔离。
  - 基于 BOS 框架建设退款标准流程，通过 SPI/扩展点开放业务流程定制、模型定制、策略定制。
  - 沉淀平台垫资、平台追缴、达人退、买贵必赔、后返、定金预售等场景能力。
  - 设计策略执行器和策略表达式，对退款单生命周期进行流程编排。
  - 梳理能力叠加冲突，按优先级解决如“定金预售 + 后返”“达人退 + 定金预售”等组合问题。
- **收益**：
  - 可以通过能力复用和叠加快速支持新行业玩法。
  - 降低业务梳理和开发成本，提高系统可维护性和变更隔离性。
  - 统一日志、错误码、异常处理机制，降低重复运维投入。

### 3. 打款系统平台化与配置化建设
- **背景**：赔付场景长期依赖客服人工打款，路径分散、容易重复赔付、无法按订单维度形成统一打款视图。
- **阶段一：平台化补齐打款能力**
  - 新增财经支付券、服务商出资打款、平台向商家打款、达人小额打款等能力。
  - 支持一次打款发多个用户资产。
  - 通过 ES 异构打款数据，支持复杂条件检索。
  - 增加跨赔付场景互斥校验，防止重复赔付。
  - 增加前置冻结/解冻商家资金能力，降低平台垫资风险。
- **阶段二：配置化接入**
  - 抽象业务流程编排、营销发券交互、财经赔付交互、客服人工赔付交互等模块。
  - 抽象 B/C 端视图层表达和用户触达配置，支持低代码接入新场景。
- **收益**：
  - 全量收敛权益类赔付、商家小额打款、客服先行赔付、物流赔付、服务保障赔付、治理规则赔付等场景。
  - 日均打款单 `75w+`、打款金额 `790w+` 元；发券 `115w+` 单/天、面额 `480w+` 元/天。
  - 平均每月新增 `10` 个赔付场景，节省产研人力 `5` 人日/月。
  - 日均拦截重复赔付 `510+` 单，显著降低平台和商家资损。

### 4. 资损防控建设
- **背景**：单纯数据一致性对账无法覆盖业务规则执行异常导致的资损，需要建设更系统的事前、事中、事后防控体系。
- **方案**：
  - 推动上下游业务规则对齐，沉淀规则文档。
  - 建设一致性对账、业务规则对账、监控预警、资金校验、攻防演练、熔断能力、资损敞口数据化运营。
  - 补齐核心业务规则对账脚本，覆盖退运费、退券/红包、权益赔付等场景。
  - 建设异常单处理工具，减少研发日常处理成本。
- **收益**：
  - 对账覆盖从 `16` 提升到 `39`。
  - 2022/2023/2024 年分别召回 `8/20/86` 个问题。
  - 实时规则校验额外召回 `5` 个线上问题。
  - Toc 出金收敛后，日均拦截 `500+` 重复打款。

### 5. 逆向交易单元化建设
- **背景**：为了支持 `VRegion 内同城多机房 + 跨 VRegion 异地多活`，逆向交易系统需要完成流量、服务、存储和业务逻辑的单元化改造。
- **方案**：
  - 建立部署决策原则、数据同步原则、服务部署类型原则、接口调度类型原则。
  - 系统性梳理 `41` 个 PSM、`8` 个 RDS、`13` 个 Redis/Abase、`71` 个 MQ、`2` 个 ES 的改单元化点。
  - 接入流量调度中间件和存储访问中间件，推进上游补 UID 等单元调度能力。
  - 针对 Redis 中心化存储问题，抽象跨单元分布式锁和跨单元计数通用方案。
  - 封装 Redis 锁 SDK：同单元直连加锁，跨单元通过 RPC 路由到正确单元，兼顾一致性和耗时。
- **收益**：
  - 完成 B/C 端单元化改造，C 端链路读写单元闭环。
  - 支持异地机房扩容与异地容灾。
  - 完成 `41` 个 PSM、`8` 个 RDS、`6` 个 Consumer、`2` 个 Redis 改造。
  - 华东放量 `30%` 后未发现改单元化相关线上问题。
