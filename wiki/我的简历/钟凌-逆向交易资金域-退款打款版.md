> 负责逆向交易资金子域建设，担任退款/打款系统 owner，主导退款系统架构与模型升级、打款系统平台化与配置化、资损防控建设以及异地多活单元化改造，持续提升资金安全、系统可维护性和业务接入效率。

> 推动退款业务逻辑、退款回调、退营销资产等能力向资金域收敛，完成模型 1.0 到 2.0 升级，通过标准流程、场景能力和领域分层支撑复杂退款场景扩展，退款/退券成功率显著提升。

> 从规则拉齐、对账、监控预警、资金校验、熔断、工具化到单元化改造，形成完整的稳定性与资损防控体系，支撑逆向交易在异地多活架构下稳定运行。

## 工作概述
- **资金子域退款/打款系统 owner**：负责业务支撑、架构升级、稳定性建设和子域规划。
- **资损防控 owner**：负责技术资损防控和业务资损防控建设。
- **逆向单元化建设 owner**：负责系统、存储、中间件改单元化设计和落地。
- **关单/退款架构升级 owner**：主导逆向交易模式升级中关单、退款活动的标准化和隔离化改造。

## 重点项目

### 1. 退款系统建设：架构 & 模型 1.0 升级
- **背景**：
  - 可退金额计算逻辑散落在多个子域服务，重复开发多、复用差、规则不一致。
  - 退款业务逻辑分散在多个服务和补偿任务里，回调链路循环依赖，维护成本和失败率高。
  - 营销退券交互不标准，退券幂等、补偿和对账都困难。
  - 旧模型字段语义混乱，难以表达聚合态退款数据和对账关系。
- **方案**：
  - 将可退金额计算结果统一由资金域提供，前端和其他子域只负责展示和消费。
  - 将退款业务逻辑收敛到资金域统一接口，财经直接回调资金域，缩短回调链路。
  - 标准化营销资产结构化数据和退还协议，实现退营销资产交互统一化。
  - 对退款单、退券单做事务落库，并增加退营销资产补偿任务。
  - 引入资金主单 + 资金子单 + 资金明细 JSON 模型：
    - 资金主单和售后单 1:1
    - 子单和下游系统数据 1:1
    - 金额明细用 JSON 扩展，适配后续新增资金类型
  - 采用 dry-run、新老模型双写、离线 diff、灰度切流等手段保障迁移平滑。
- **收益**：
  - 退款和退券成功率显著提升。
  - 用户因未到账、未退券、数据不一致引发的技术类进线归零。
  - 单个需求评估与开发节省至少 `2pd`。
  - 新模型可支撑未来 `3-5` 年业务扩展，如退金币、余额等。

### 2. 退款系统建设：架构 2.0 升级 / 交易模式升级
- **背景**：
  - 平台与业务、业务与业务之间耦合严重，流程中大量 if 分支交错，复杂度高。
  - 业务隔离差，需求改动影响面大，风险不可控。
  - 仓储层面向过程实现，查询方法膨胀，PO 直接暴露给业务。
- **方案**：
  - 定义标准化业务身份，作为业务隔离的唯一标识。
  - 架构分为三层：
    - 基础域 / 业务域
    - 标准业务能力
    - 基于场景能力或业务身份的扩展层
  - 基于 BOS 框架实现退款标准流程，并开放业务流程、扩展点、模型三类定制能力。
  - 沉淀平台垫资、平台追缴、达人退、买贵必赔、后返、定金预售等场景能力，支持复用和叠加。
  - 建立资金基础域，通过共享模型（SDO）、领域模型（DO）和仓储防腐层提供稳定可复用的 CRUD 能力。
- **收益**：
  - 复杂业务得到结构化拆分，理解和维护成本下降。
  - 业务隔离增强，平台风险更可控。
  - 新业务可通过能力复用和叠加快速接入。

### 3. 打款系统建设：平台化到配置化
- **背景**：
  - 用户赔付长期依赖客服人工打款，效率低。
  - 打款路径分散，容易与售后、客服等链路重复赔付，造成资损。
  - 无法按订单维度沉淀统一打款视图，数据收敛和对账困难。
- **方案**：
  - 以“退款 + 打款”一体化的逆向资金能力为目标，统一沉淀用户、商家、平台视角的出金能力。
  - **阶段一：平台化**
    - 抽象打款流程
    - 补齐多出资方、多资产形式的打款能力
    - 增加打款管控能力和 B/C 端视图表达能力
  - **阶段二：配置化**
    - 将强关联赔付场景的核心逻辑抽象为配置模块
    - 业务流程编排模块包括：业务规则校验、多段单调度、营销发券交互、财经赔付交互、客服赔付交互
    - 视图层配置包括：C 端通知和 B/C 端赔付去向展示
- **收益**：
  - 平均每月新增 `10` 个赔付场景，节省产研人力 `5` 人日/月。
  - 打款系统解决跨业务方重复赔付问题后：
    - 百万订单商家人工 CPO 下降 `18%`（`21.93 -> 17.88`）
    - 商家资金流失率下降近 `22%`（`-1.01% -> 0.88%`）

### 4. 资损防控建设
- **背景**：随着平台交易量增长，逆向交易资损问题直接影响资金安全、用户体验和商家满意度，需要形成系统化防控能力。
- **方案**：
  - **技术防控**：一致性建设、风险左移、架构升级
  - **业务防控**：数据洞察、成本监控
  - 已完成的体系化动作覆盖：
    - 事前：上下游规则拉齐、鲁棒性设计规约、资损点梳理、攻防演练
    - 事中：一致性对账、监控预警、资金校验、熔断能力、资损敞口数据化运营、出金链路收敛
    - 事后：部分外呼追偿能力仍待建设
  - 通过售后小工具提供 `10+` 异常单处理工具，降低研发介入成本。
- **收益**：
  - 2022 年逆向整体召回率 `61% (8/13)`；体系化建设启动后当年后续问题召回率 `100% (8/8)`。
  - 2023 年无逆向主责资损问题，额外召回其他领域 `5` 个问题。
  - Toc 出金收敛后，日均拦截 `10+` 重复打款。
  - 通过资金异步校验召回 `5+` 异常数据 case，通过一致性对账召回 `45+` 数据不一致 case。

### 5. 单元化建设
- **背景**：为支持 `VRegion 内同城多机房 + 跨 VRegion 异地多活`，逆向交易系统需要进行流量、服务、存储、缓存和 MQ 的全面改单元化。
- **方案**：
  - 梳理并定义部署原则：
    - 是否华东部署
    - 数据同步策略
    - 服务部署类型
    - 接口调度类型
  - 接入流量调度中间件和存储访问中间件。
  - 制定改单元化 checklist，系统性梳理资源：
    - `41` 个 PSM
    - `8` 个 RDS
    - `13` 个 Redis / Abase
    - `71` 个 MQ
    - `2` 个 ES
  - 解决 Redis 中心化存储改造难点：
    - **分布式锁**：选择“Redis 分单元部署 + 单元调度接口加锁”的方案；同单元直连，跨单元走 RPC，兼顾一致性和 RT。
    - **计数能力**：对于全局累计赔付金额等场景，使用中心调度接口统一写中心 Redis，实现全局计数。
- **收益**：
  - 完成 `41` 个 PSM 中间件接入和适配。
  - 完成 `8` 个 RDS 单元同步。
  - 完成 `6` 个 Consumer 逻辑下沉改造。
  - 完成 `2` 个 Redis 中心化存储改造。
  - 华东放量 `30%` 后未发现改单元化相关线上问题。
