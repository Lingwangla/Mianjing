3. 怎么解决少卖？

### **核心思想**：
 1. 能挡在前面的绝不放后面；
 2. 能缓存的不查数据库；
3. 能异步的不同步；
4. 能快速失败的不排队；
5. 宁可少卖，不可超卖；
6. 最终一致，不追求强一致。

1. localcache本地缓存
2. 将热key拆分多个key，通过redis cluster集群部署，流量分摊多个redis节点，降低单点实例问题
3. 防止缓存击穿，回源db/下游，造成大量瞬时写redis，则增加分布式锁
#### 2. Redis + MQ + MySQL 如何保证库存最终一致？
##### 异常流程
```
Redis成功
MQ丢了

↓

Redis少1
DB没少1

↓

库存不一致
```

##### 最终一致性的终极方案

采用**Redis + RocketMQ事务消息 + MySQL + Binlog + 对账系统
```
用户

↓

Redis预扣库存

↓

RocketMQ事务消息

↓

订单服务

↓

MySQL

↓

binlog

↓

Canal

↓

库存对账服务

↓

修正Redis库存
```

秒杀库存的一致性不追求强一致，而追求最终一致。

Redis负责高并发预扣库存，RocketMQ负责削峰异步化，MySQL作为最终权威库存。

为了避免 Redis 扣减成功但消息丢失，可以使用 RocketMQ 事务消息保证 Redis 与 MQ 的一致性。

消费阶段通过幂等设计避免重复扣减，通过重试机制和死信队列保证消息最终被消费。

如果订单创建最终失败，则通过库存流水表 + 定时补偿任务恢复 Redis 库存。

最后通过 Binlog + 对账系统进行全链路校准，实现库存最终一致。

#### 3. 既然前置 Token 已经只有 100 人进入了，为什么还需要 Redis Lua 扣库存？
前置 Token 的作用是流量过滤，把 100w 请求缩减到 100 个请求，它解决的是 Redis Hot Key 和系统压力问题；

Redis Lua 扣库存解决的是真实库存状态问题，因为 Token 本身不具备事务、回滚和补偿能力。

用户拿到 Token 后可能退出、超时、下单失败，如果没有 Redis 库存层，就会出现库存丢失、少卖的问题。

因此在生产环境中，Token 和 Redis Lua 是两层控制：

Token 负责控制谁能进入；

Redis 库存负责决定谁真正抢到商品；

Token 只是资格，库存才是真正的资源。
