# 设计模式划分
软件工程设计模式分类总览，经典的 **GoF（Gang of Four）23种设计模式**，可以按照关注点分为三大类：
```
设计模式
├── 一、创建型模式（5种）
│     ├── 单例模式
│     ├── 工厂方法模式
│     ├── 抽象工厂模式
│     ├── 建造者模式
│     └── 原型模式
│
├── 二、结构型模式（7种）
│     ├── 适配器模式
│     ├── 桥接模式
│     ├── 组合模式
│     ├── 装饰器模式
│     ├── 外观模式
│     ├── 代理模式
│     └── 享元模式
│
└── 三、行为型模式（11种）
      ├── 观察者模式
      ├── 策略模式
      ├── 模板方法模式
      ├── 责任链模式
      ├── 命令模式
      ├── 状态模式
      ├── 迭代器模式
      ├── 中介者模式
      ├── 备忘录模式
      ├── 解释器模式
      └── 访问者模式
```

1. 创建型模式（解决“对象如何创建”）
2. 结构型模式（解决“对象如何组织”）
3. 行为型模式（解决“对象如何协作”）


# Golang + 电商 + 微服务 + DDD 场景下 8 大设计模式笔记

## 一、设计模式总体分类

在大型电商系统中，以下 8 种设计模式出现频率最高：

- 策略模式（Strategy）
- 责任链模式（Chain Of Responsibility）
- 观察者模式（Observer）
- 装饰器模式（Decorator）
- 代理模式（Proxy）
- 状态模式（State）
- 模板方法模式（Template Method）
- 适配器模式（Adapter）

它们几乎贯穿：

- 订单中心
- 营销中心
- 库存中心
- 支付中心
- 逆向交易中心
- 用户中心
- 消息中心

以及：

- Gin
- GORM
- gRPC
- RocketMQ
- Redis
- DDD

---

# 1. 策略模式（Strategy）

## 核心思想

> 将算法封装起来，使算法可以自由切换。

### 解决的问题

避免：

```go
if vip {
    ...
} else if coupon {
    ...
} else if fullReduce {
    ...
}
```

大量 if-else。

强调：

> 组合优于继承

---

## UML

```text
            Strategy
                ↑
      -------------------
      ↓                 ↓
Discount          FullReduce
                ↑
             Context
```

---

## 业务场景：营销优惠计算

支持：

- 满减
- 优惠券
- VIP折扣
- 新人首单

统一接口：

```go
type DiscountStrategy interface {
    Calc(order Order) Money
}
```

实现：

```go
VipDiscount

CouponDiscount

FullReduceDiscount
```

运行时动态切换：

```go
strategyFactory.Get(userType)
```

### DDD位置

Domain Service

---

## 框架源码场景：gRPC负载均衡

支持：

- pick_first
- round_robin

统一：

```go
Picker
```

源码目录：

```text
grpc-go/balancer
```

---

# 2. 责任链模式（Chain Of Responsibility）

## 核心思想

> 请求沿着链传递，每个节点决定是否处理。

### 强调

流程解耦。

---

## UML

```text
Handler1
    ↓
Handler2
    ↓
Handler3
    ↓
Business
```

---

## 业务场景：风控系统

下单流程：

```text
黑名单校验
↓

设备风险

↓

账号风险

↓

支付风险

↓

人工审核
```

统一：

```go
type RiskHandler interface {
    Handle()
}
```

通过：

```go
handler.next
```

形成责任链。

### DDD位置

Domain Pipeline

---

## 框架源码场景：Gin Middleware

源码：

```go
engine.Use()
```

内部：

```go
c.Next()
```

形成：

```text
Recovery

↓

Trace

↓

Log

↓

Auth

↓

Business
```

源码：

```go
gin.Context.Next()
```

# 八大设计模式汇总

| 模式 | 设计理念 | 业务系统应用 | 框架源码应用 | DDD位置 |
|------|---------|---------|---------|---------|
| Strategy（策略） | 算法可切换，组合优于继承 | 营销优惠、路由、限流 | gRPC LoadBalance | Domain Service |
| Chain（责任链） | 请求沿链传递 | 风控审核、审批流 | Gin Middleware | Domain Pipeline |
| Observer（观察者） | 发布订阅，一对多通知 | 支付成功事件、库存扣减 | etcd Watch、RocketMQ | Domain Event |
| Decorator（装饰器） | 动态增强能力 | DAO缓存、trace、metric | Gin Middleware | Infrastructure |
| Proxy（代理） | 控制访问、透明调用 | RPC调用、远程服务 | gRPC Stub | ACL |
| State（状态） | 状态决定行为 | 订单状态机、售后状态机 | TCP状态机 | Aggregate Root |
| Template（模板方法） | 流程固定，步骤扩展 | 支付流程、退款流程 | GORM Transaction | Application Service |
| Adapter（适配器） | 接口兼容 | 多支付渠道、物流渠道 | database/sql driver | ACL |

---

# 八种模式协同工作

一个典型的订单支付成功链路：

```text
Order Aggregate
       │
       ▼
状态模式（State）
       │
       ▼
模板方法模式（Template）
       │
       ▼
适配器模式（Adapter）
   微信支付 / 支付宝
       │
       ▼
代理模式（Proxy）
RPC调用渠道服务
       │
       ▼
观察者模式（Observer）
发送 OrderPaidEvent
       │
       ▼
责任链模式（Chain）
消费端风控处理
       │
       ▼
策略模式（Strategy）
营销规则计算
       │
       ▼
装饰器模式（Decorator）
trace + metric + cache
```

## 本质

大型系统并不是只使用某一种设计模式，而是：

> 多种设计模式协同工作，共同支撑复杂业务。

这是 DDD + 微服务 + 电商架构中最常见的设计方式，也是高级工程师和架构师面试最关注的能力。
