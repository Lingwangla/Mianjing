# 为什么互联网大厂把 MySQL 隔离级别设为 RC

来源：[知乎回答](https://www.zhihu.com/question/649997016/answer/1946250160041616784)

## 一句话结论

MySQL 默认是 `RR`，核心是历史上要兼容早期 `statement` 格式的 `binlog`，避免主从复制出现数据不一致；而互联网大厂常改成 `RC`，核心是为了减少 `Gap Lock/Next-Key Lock` 带来的锁冲突和死锁，提升高并发场景下的吞吐与响应时间。  

前提是：`binlog` 不能再用 `statement`，通常要用 `row` 或 `mixed`。

## 1. RC 和 RR 的核心区别

### RC：Read Committed，读已提交

- 只能读到其他事务已经提交的数据
- 解决脏读
- 不能解决不可重复读
- 也可能出现幻读
- 锁更轻，一般以行锁为主

### RR：Repeatable Read，可重复读

- 同一个事务内，多次读取同一批数据，结果尽量保持一致
- 解决不可重复读
- InnoDB 里通过 MVCC + Gap Lock/Next-Key Lock 尽量避免幻读
- 一致性更强，但锁开销更大

## 2. 为什么 MySQL 默认是 RR

这件事不能只看事务隔离级别，还要结合 MySQL 的主从复制来看。

### 2.1 主从复制怎么工作的

主从复制大致分 3 步：

1. 主库把更新操作写入 `binlog`
2. 从库把主库的 `binlog` 拉到本地，写入 `relay log`
3. 从库重放 `relay log` 中的操作

### 2.2 binlog 有三种格式

- `statement`：记录 SQL 原文
- `row`：记录每一行数据怎么改了
- `mixed`：两者混用

其中：

- `statement` 日志量小，但依赖 SQL 在从库再次执行后还能得到与主库一致的结果
- `row` 更稳，因为它记录的是最终修改结果，不依赖从库重新按条件筛数据

### 2.3 RR 默认值的历史原因

早期 MySQL 主要依赖 `statement` 格式复制。

如果此时隔离级别用 `RC`，并发事务在主库和从库上的执行顺序一旦不同，就可能导致：

- 主库结果是 A
- 从库根据 SQL 原文重放后结果变成 B

也就是主从不一致。

而 `RR` 下由于有 `Gap Lock/Next-Key Lock`，会把某些并发更新挡住，减少这类因为并发可见性不同带来的复制歧义，所以 MySQL 默认选了 `RR`。

## 3. 为什么互联网大厂更常用 RC

### 3.1 根因是并发更高

`RR` 为了更强一致性，会引入：

- `Gap Lock`
- `Next-Key Lock`

这些锁能减少幻读，但副作用也很明显：

- 锁冲突更多
- 死锁概率更高
- 高并发写入时等待更久
- 吞吐更差

而 `RC` 下：

- 不依赖 Gap Lock 去保证可重复读
- 锁范围更小
- 冲突更少
- 事务更容易快速提交

所以在高并发互联网业务里，`RC` 常常更符合目标：先把整体性能和稳定性做上去。

### 3.2 适合 RC 的典型场景

- 电商高并发下单
- 秒杀
- 高频更新的交易类系统
- 对吞吐和延迟更敏感的在线业务

### 3.3 RR 更适合的场景

- 强调一致性的核心账务场景
- 库存这类对并发异常更敏感的业务
- 业务上确实依赖“事务内重复读一致”的场景

## 4. 把隔离级别改成 RC 后要注意什么

### 4.1 要接受幻读和不可重复读风险

`RC` 不保证事务内多次查询一定一致，因此：

- 同一事务前后两次读结果可能不同
- 范围查询可能多出或少掉记录

很多互联网业务并不依赖“长事务内反复读取必须完全一致”，所以可以接受。

如果确实不能接受，需要额外补偿：

- 用显式锁，例如 `select ... for update`
- 缩短事务
- 在业务侧做幂等、校验、重试

### 4.2 binlog 不能再用 statement

这是最关键的配套条件。

如果数据库已经切到 `RC`，`binlog` 仍然使用 `statement`，就又会回到早年那个主从不一致的问题。

因此一般要求：

- 优先使用 `row`
- 或使用 `mixed`

## 5. 面试回答模板

如果面试官问：“MySQL 默认是 RR，为什么很多大厂反而配成 RC？”

可以直接这样答：

> MySQL 默认用 RR，主要是历史上要兼容早期 statement binlog 的主从复制，避免并发事务在主从重放时产生数据不一致。  
> 但互联网大厂更看重高并发性能，RR 会引入 Gap Lock 和 Next-Key Lock，锁冲突和死锁概率更高。  
> 所以很多公司会改成 RC，减少锁范围，提高吞吐。  
> 不过改成 RC 的前提是 binlog 不能再用 statement，一般要配成 row 或 mixed，同时业务上要能接受不可重复读和幻读，或者通过显式锁和业务校验补上。

## 6. 这道题真正想考什么

这题本质上不是只考“隔离级别定义”，而是在考你能不能把下面几件事串起来：

- 事务隔离级别
- MVCC 和锁
- Gap Lock / Next-Key Lock
- binlog 三种格式
- 主从复制一致性
- 一致性和性能之间的工程取舍

## 7. 一个更工程化的理解

默认值是“历史兼容 + 保守一致性”的选择。  
大厂改成 `RC` 是“已知业务特点后的主动取舍”：

- 放弃一部分默认一致性保证
- 换取更高并发性能
- 再用 `row binlog + 业务兜底` 把风险收回来

这就是数据库参数为什么不能只背定义，必须结合业务场景来看。
